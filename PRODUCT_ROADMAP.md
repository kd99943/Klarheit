# Klarheit 产品路线图

> 定位：高端 D2C 眼镜电商平台，核心差异化功能为 AR 虚拟试戴。
> 目标：从当前 Demo 状态推进至可正式发布的生产级产品。

---

## 当前状态评估

| 模块 | 状态 | 说明 |
|------|------|------|
| 用户注册/登录 | ✅ 基本可用 | JWT 认证已通，但安全配置需加固 |
| 产品目录 | ✅ 基本可用 | 4 款产品已硬编码，缺过滤/搜索 |
| 配镜向导 | ⚠️ 不完整 | 步骤 UI 存在，但状态未持久化 |
| AR 虚拟试戴 | ❌ 空壳 | 仅有占位 UI，无实际功能 |
| 结账流程 | ⚠️ 不完整 | 表单存在，支付未接入，无订单确认 |
| 订单管理 | ❌ 缺失 | 我的账户页无真实数据 |
| 安全与稳定性 | ❌ 不达标 | 硬编码密钥、无测试、无日志 |
| 部署基础设施 | ❌ 缺失 | 无 Docker、无 CI/CD |

---

## 阶段规划

### Phase 0 — 地基加固（1~2 周）

**目标：** 在添加任何新功能之前，先把现有代码的安全和质量问题修复，避免技术债务在后续阶段放大。

**任务清单：**

#### 安全
- [ ] 将 JWT 密钥迁移至环境变量，从代码库中移除硬编码值
- [ ] 限制 CORS 白名单，仅允许前端域名
- [ ] 对认证接口添加速率限制（防暴力破解）
- [ ] 对所有用户输入添加后端校验（处方数值范围、邮箱格式等）

#### 代码质量
- [ ] 移除处方表单中的硬编码测试默认值（-2.25 等）
- [ ] 统一前端 API 错误处理，添加全局 Error Boundary
- [ ] 提取重复的产品加载逻辑为共享 hook（`useProducts`）
- [ ] 后端添加结构化日志（SLF4J + Logback）

#### 测试
- [ ] 后端：补充核心业务逻辑的单元测试（订单计算、处方校验）
- [ ] 前端：为关键组件添加基础渲染测试

#### 数据库
- [ ] 引入 Flyway 管理数据库迁移，替代 `schema.sql`

**阶段验收测试：**
- 安全扫描：无硬编码密钥泄露
- 所有现有后端集成测试通过
- 前端无控制台报错，所有页面正常渲染

---

### Phase 1 — 核心用户流程闭环（2~3 周）

**目标：** 让用户能完整走完"浏览 → 配镜 → 下单 → 确认"的核心购买链路。

**任务清单：**

#### 配镜向导（Config Lab）修复
- [ ] 修复步骤间状态传递（处方数据 → 镜片选择 → 汇总）
- [ ] 添加处方数值的前端实时校验（SPH/CYL 范围、AXIS 0-180°）
- [ ] 镜片选择步骤与后端 `lens-options` API 真实联动
- [ ] 配镜完成后携带完整数据跳转结账页

#### 结账流程完善
- [ ] 结账页接收并展示来自配镜向导的完整数据
- [ ] 订单提交后展示订单确认页（含订单号、商品摘要、预计交期）
- [ ] 接入邮件服务（推荐 Resend 或 SendGrid），发送订单确认邮件

#### 我的账户 — 订单历史
- [ ] 后端添加 `GET /api/v1/orders/my` 接口，返回当前用户订单列表
- [ ] 前端订单历史页展示真实数据（订单号、状态、金额、日期）
- [ ] 订单详情页（点击订单可查看完整信息）

**阶段验收测试：**
- 完整走通注册 → 浏览产品 → 配镜 → 下单 → 收到确认邮件 → 在账户页查看订单
- 处方输入错误值时前后端均有拦截提示

---

### Phase 2 — AR 虚拟试戴（3~4 周）

**目标：** 实现核心差异化功能，让用户在购买前能虚拟试戴眼镜。

**技术选型建议（三选一）：**

| 方案 | 技术 | 优点 | 缺点 |
|------|------|------|------|
| A（推荐） | MediaPipe FaceMesh + Three.js | 开源免费，可完全自定义，无第三方依赖 | 开发周期较长，需要 3D 建模 |
| B | Tryon.io / Banuba SDK | 开箱即用，效果成熟 | 有授权费用，依赖第三方服务 |
| C | 静态图片叠加（简化版） | 开发最快，无需 3D | 体验较差，不是真正的 AR |

**推荐方案 A 的任务清单：**

- [ ] 为 4 款产品制作或采购 3D 眼镜模型（.glb 格式）
- [ ] 集成 MediaPipe FaceMesh 实现实时人脸关键点检测
- [ ] 使用 Three.js 将 3D 眼镜模型叠加到视频流上
- [ ] 实现眼镜位置自动对齐（根据瞳距、鼻梁位置）
- [ ] AR 页面添加截图/分享功能
- [ ] 移动端摄像头权限处理和适配
- [ ] 添加"无摄像头"降级方案（静态图片试戴）

**阶段验收测试：**
- 在 Chrome/Safari 桌面端和移动端均可正常启动摄像头
- 眼镜模型能跟随头部移动保持稳定叠加
- 切换不同产品时模型实时更新
- 无摄像头时降级方案正常展示

---

### Phase 3 — 支付与商业化（2~3 周）

**目标：** 接入真实支付，让平台具备实际收款能力。

**任务清单：**

#### 支付集成
- [ ] 确定支付方案（国内：微信支付/支付宝；海外：Stripe）
- [ ] 后端实现支付订单创建接口
- [ ] 后端实现支付回调（Webhook）处理，更新订单状态
- [ ] 前端集成支付 SDK，替换当前的假表单
- [ ] 支付成功/失败页面

#### 订单状态管理
- [ ] 订单状态流转：`PENDING_PAYMENT` → `PAID` → `PROCESSING` → `SHIPPED` → `DELIVERED`
- [ ] 状态变更时发送邮件通知用户
- [ ] 后台管理接口（更新订单状态，供运营人员使用）

**阶段验收测试：**
- 使用沙箱/测试模式完成一笔完整支付
- 支付成功后订单状态正确更新
- 用户收到支付确认邮件

### Phase 4 — 产品体验提升（持续迭代）

**目标：** 提升产品完整度和用户体验，为正式发布做准备。

**任务清单：**

#### 产品目录增强
- [ ] 产品过滤（按材质、价格区间、镜框形状）
- [ ] 产品搜索
- [ ] 产品详情页（独立 URL，SEO 友好）

#### 用户体验

- [ ] 最近浏览记录
- [ ] 优惠码/折扣功能
- [ ] 用户地址簿（保存常用收货地址）

#### 性能
- [ ] 产品列表接口添加分页
- [ ] 前端添加图片懒加载
- [ ] 后端接口添加缓存（Redis 或 Spring Cache）

---

### Phase 5 — 生产部署（1~2 周）

**目标：** 将应用部署到生产环境，具备上线条件。

**任务清单：**

#### 容器化
- [ ] 编写 `Dockerfile`（前端 + 后端分别）
- [ ] 编写 `docker-compose.yml`（本地一键启动）

#### CI/CD
- [ ] 配置 GitHub Actions：代码推送自动运行测试
- [ ] 配置自动构建和部署流水线

#### 生产配置
- [ ] 配置生产环境变量（数据库、JWT 密钥、邮件服务）
- [ ] 配置 HTTPS（SSL 证书）
- [ ] 配置域名和 CDN（静态资源加速）
- [ ] 数据库备份策略

#### 监控
- [ ] 接入错误监控（Sentry）
- [ ] 配置基础性能监控（响应时间、错误率）

**阶段验收测试：**
- 生产环境完整走通核心购买链路
- HTTPS 正常，无混合内容警告
- 错误监控收到测试告警

---

## 优先级总结

```
Phase 0（地基）→ Phase 1（流程闭环）→ Phase 2（AR 核心功能）
     → Phase 3（支付）→ Phase 4（体验提升）→ Phase 5（部署上线）
```

**关键路径：** Phase 0 和 Phase 1 是所有后续工作的前提，必须先完成。Phase 2 是核心差异化，应在支付之前完成，因为 AR 体验是用户选择 Klarheit 而非竞品的核心理由。

## 技术债务备忘

以下问题已知但未纳入阶段任务，需在合适时机处理：

- 前端缺少 API 响应类型的完整 TypeScript 定义
- 后端缺少 OpenAPI/Swagger 文档
- 产品数据目前通过 `DataInitializer` 硬编码，后续需要管理后台或 CMS
- 无库存管理系统（当前假设无限库存）
