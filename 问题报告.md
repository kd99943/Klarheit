  Klarheit 项目全面审查报告

  🔴 严重问题 (Critical)

  1. .env 文件已提交到 Git — 真实密码泄露

  backend/.env 包含真实 MySQL 密码 (Fh050806)、JWT 密钥，且已被 Git 跟踪。虽然 .gitignore 列出了
  backend/.env，但文件在被加入 ignore 之前就已经被 commit 了，所以它仍然在版本历史中。

  修复： 从 Git 历史中移除该文件（git rm --cached backend/.env），然后 rotate 所有泄露的密钥。

  2. 支付回调端点无鉴权 — 任何人可伪造支付成功

  PaymentCallbackController.java:73-83 的 /payments/callback/mock-trigger 是 permitAll() 的，任何人可以发送一个 POST
  请求直接将任意订单标记为 PAID：

  @PostMapping("/mock-trigger")
  public ResponseEntity<String> mockTrigger(@RequestBody Map<String, String> request) {
      // 没有任何身份验证或签名验证
      orderService.completePayment(orderNumber, txId, channel);
  }

  /orders/{orderNumber}/status 同样是公开的，任何人可以查询任何订单状态。

  修复： mock-trigger 端点至少应该加 @Profile("local") 限制，只在开发环境可用。生产环境应该移除或加鉴权。

  3. XSS 注入风险 — Alipay Form 直接拼接用户输入

  PaymentServiceImpl.java:60-66 的 buildMockAlipayForm 直接将 orderNumber 和 amount 拼接进 HTML：

  return "<input type='hidden' name='out_trade_no' value='" + orderNumber + "'/>";

  如果 orderNumber 包含恶意 HTML/JS，会被直接执行。

  4. docker-compose.yml 硬编码数据库密码

  root_security_pass_2026 和 app_secure_pass_klarheit 明文写在文件中。应该用环境变量或 Docker secrets。

  🟠 架构与数据模型问题

  5. Flyway 迁移与 JPA 实体严重不一致

  - V1 创建了 lens_option_types VARCHAR 列，但 Order 实体使用 @ManyToMany 关系（join table order_lens_options）。V1 的
    lens_option_types 列成了孤儿列，没有任何迁移移除它。
  - V1 的 prescriptions 表有 user_email 列，V3 才移除。
  - Order.finishId 字段存在于实体中，但从未出现在任何 Flyway 迁移中。
  - Product 实体有 name_en, name_zh, material_en, material_zh，但 V1 只有 name 和 material。这些列的添加没有对应迁移（靠
    ddl-auto: update 补的）。

  根本原因： application-local.yml 设置 ddl-auto: update，让 Hibernate 自动修补 schema，绕过了 Flyway。这意味着 Flyway
  迁移和实际 schema 已经脱节。生产环境如果用 ddl-auto: validate 会直接崩溃。

  6. DataInitializer 做了 Flyway 应该做的事

  产品数据和镜片选项数据的 seed 逻辑写在 Java 代码中（DataInitializer.java），而不是 Flyway 迁移中。这意味着：
  - 无法追踪数据变更历史
  - 无法在不同环境间重现一致的初始数据
  - 与 Flyway 的 baselineOnMigrate 配置冲突

  7. 名称不一致 — "Lumina" vs "Klarheit"

  项目中到处混用旧名：
  - Cookie 名：lumina_auth_token（应为 klarheit_auth_token）
  - 数据库名：lumina_optics（本地 .env）vs klarheit（docker-compose）
  - 代码包名已经是 com.klarheit.backend，但基础设施残留旧名

  8. Product 实体有冗余字段

  name 和 material 字段与 nameEn/materialEn 完全重复。应该移除 name 和 material，或者将它们作为 nameEn 的 fallback。

  9. LensOption 硬编码映射

  LensOptionService.java 用 switch 硬编码了
  categoryFor、labelFor、descriptionFor。这些应该存在数据库中，否则每次新增镜片类型都要改代码。

  🟡 后端问题

  10. Alipay Mock 收银台硬编码 localhost:5173

  PaymentCallbackController.java:226-237 中的 JS 硬编码了：
  window.location.href = 'http://localhost:5173/my-account';
  window.location.href = 'http://localhost:5173/checkout';
  生产环境无法使用。

  11. application-local.yml 硬编码 JWT fallback 密钥

  secret: ${APP_JWT_SECRET:ZGV2LWxvY2FsLWp3dC1zZWNyZXQta2xhcmhlaXQtMjAyNi0wNS0xMC1yb3RhdGUtbWU=}
  如果环境变量未设置，任何人可以用这个公开的默认密钥伪造 JWT。

  12. 微信支付回调未实际处理订单

  handleWeChatCallback 方法（第 63-69 行）只返回了一个固定的 XML success，没有调用
  orderService.completePayment()。微信支付永远不会真正更新订单状态。

  13. 中英文错误信息混用

  CouponService 的异常消息全部是中文（"优惠码不能为空。"），而其他 Service 都是英文。应该统一用 i18n key 或统一语言。

  14. EmailService 使用 %% 导致格式化问题

  EmailService.java:54 的 width: 100%% 中的 %% 在 String.formatted() 中会被转义为 %，这是正确的。但 HTML 中的 ©
  在某些编码环境下可能显示异常，应该用 &copy;。

  15. Rate Limiter 不适合生产

  - 内存 Map 存储，无法水平扩展
  - LinkedHashMap 的 removeEldestEntry 在 synchronizedMap 包装下不是线程安全的（LRU access-order 的 get
    操作会修改内部链表）
  - 没有对 X-Forwarded-For 头做防伪造处理（攻击者可以随意设置 IP）

  🟡 前端问题

  16. vite 重复依赖

  package.json 中 vite 同时出现在 dependencies 和 devDependencies。应该只在 devDependencies。

  17. 不需要的依赖

  - express — React 前端不需要 Express
  - dotenv — Vite 原生支持 .env 文件，不需要 dotenv 包
  - @google/genai — 代码中没有任何地方使用 Gemini API

  18. Product 接口缺少本地化字段

  api.ts 中的 Product 接口只有 name 和 material，没有 nameEn, nameZh, materialEn,
  materialZh。但后端返回了这些字段，前端页面（CollectionsPage, Checkout）也在使用它们。TypeScript 没有报错只是因为用了
  as 断言或绕过了类型检查。

  19. 汇率硬编码

  utils.ts:13 的 formatPrice 硬编码了 amount * 7.3 的汇率。这个汇率会波动，应该从 API 获取或配置化。

  20. Checkout 页面错误信息混合中英文

  Checkout.tsx:199：
  errors.push(`${range.name} 必须是有效数字 (must be a valid number).`);
  应该用 i18n 翻译。

  21. front_end/dist/ 已提交到 Git

  构建产物不应该在版本控制中，.gitignore 已列出但目录已被跟踪。

  22. CSS-in-JS 在 JSX 中

  Checkout.tsx:482-488 在 JSX 中直接插入 <style> 标签定义动画。应该用 Tailwind 的 animate- 工具类或 CSS 文件。

  🔵 小问题 & 改进建议

  23. Order 实体缺少 @PrePersist 的 createdAt 设置

  虽然有 @PrePersist，但如果 JPA provider 不保证在 persist 前调用，可能会有 null 值。建议加 nullable = false
  的数据库默认值（DEFAULT CURRENT_TIMESTAMP）。

  24. 缺少分页

  ProductRepository.findAll() 返回所有产品，OrderRepository.findByUserIdOrderByCreatedAtDesc()
  返回用户所有订单。数据量大了会有性能问题。

  25. 没有 OpenAPI/Swagger 文档

  PRODUCT_ROADMAP 也提到了这个技术债务。

  26. config-lab 的 LensOption category 字段

  前端 LensOption 接口有 category 字段，但后端 LensOption 实体没有。这是 LensOptionService 硬编码的 DTO
  字段。如果将来新增镜片类型，前端分类逻辑会失效。

  27. 缺少 favicon 和 meta tags

  index.html 可能缺少 SEO 相关的 meta 标签和 favicon。

  28. 测试覆盖不完整

  虽然有测试文件，但很多关键路径（支付回调、优惠券并发、处方校验边界）的测试覆盖不足。
